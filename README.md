# 보상 트랜잭션 기반 데이터 정합성 보장 설계

## 1. 프로젝트 개요

분산 트랜잭션 환경에서 일부 작업 실패 시  
선행 작업이 커밋되어 데이터 정합성이 깨지는 문제를 해결하기 위해  
보상 트랜잭션 기반의 롤백 전략을 설계했다.

---

## 2. 문제 정의

### 2.1 기존 구조

하나의 요청에서 다음과 같은 작업이 순차적으로 실행되는 구조였다.

1. 주문 생성
2. 포인트 차감
3. 쿠폰 사용
4. 메시지 발행

각 작업은 개별 트랜잭션으로 커밋되며  
후행 작업 실패 시 이전 작업을 롤백할 수 없는 구조였다.

---

### 2.2 실제 장애 상황

주문 생성 이후 포인트 차감 실패

→ 주문은 생성되었지만 포인트는 유지됨  
→ 데이터 정합성 붕괴  
→ 수동 데이터 보정 필요

---

### 2.3 기존 방식의 한계

- DB 트랜잭션으로는 이미 커밋된 작업 롤백 불가능
- 장애 대응을 위한 운영 개입 필요
- 재처리 시 중복 실행 발생

---

## 3. 목표

- 데이터 정합성 보장
- 장애 발생 시 자동 복구
- 재처리 가능 구조 설계
- 멱등성 보장

---

## 4. 설계 전략

### 4.1 보상 트랜잭션 패턴 적용

각 작업을 다음과 같이 정의했다.

| Step | Action | Compensation |
|------|--------|--------------|
Order | 주문 생성 | 주문 취소 |
Point | 포인트 차감 | 포인트 복구 |
Coupon | 쿠폰 사용 | 쿠폰 복원 |

실패 발생 시 성공한 Step을 역순으로 보상 실행하도록 구성했다.

---

### 4.2 상태 기반 트랜잭션 관리

각 Step의 상태를 저장하도록 설계했다.

```
READY → SUCCESS → FAILED → COMPENSATED
```

이를 통해

- 재시도 가능
- 중복 실행 방지
- 장애 복구 자동화

를 달성했다.

---

### 4.3 트랜잭션 오케스트레이션

Application Layer에서 전체 흐름을 제어하도록 구성했다.

```
executeStep()
↓
상태 저장
↓
실패 발생
↓
완료된 Step 조회
↓
보상 트랜잭션 실행
```

---

## 5. 기술적 의사결정

### 5.1 Choreography 대신 Orchestration 선택 이유

Choreography 방식은

- 흐름 파악이 어렵고
- 장애 추적이 복잡

하다는 단점이 있어  
중앙에서 제어 가능한 Orchestration 방식을 선택했다.

---

### 5.2 상태 저장 위치

DB에 저장하도록 설계

이유:

- 장애 복구 시점에도 상태 유지
- 운영 환경에서 추적 가능

---

## 6. 트랜잭션 흐름

### 정상 흐름

1. 주문 생성 SUCCESS
2. 포인트 차감 SUCCESS
3. 쿠폰 사용 SUCCESS

---

### 실패 흐름

포인트 차감 실패

→ 주문 생성 COMPENSATE 실행

---

## 7. 트러블슈팅

### 7.1 보상 트랜잭션 중복 실행

#### 문제

네트워크 타임아웃 이후 재시도 과정에서  
보상 로직이 중복 실행됨.

#### 분석

로그 분석 결과

- 동일한 Step에 대해 두 번의 보상 요청 발생
- 상태 검증 로직 부재

#### 해결

보상 실행 전 상태 검증

```
if (stepStatus != SUCCESS) return;
```

멱등성 보장.

---

### 7.2 재처리 시 중복 실행

#### 문제

장애 복구 후 재시도 시  
이미 성공한 Step이 다시 실행됨.

#### 해결

상태 기반 실행 분기 처리

```
if (stepStatus == SUCCESS) skip;
```

---

### 7.3 보상 순서 문제

#### 문제

보상이 실행될 때 실행 순서가 보장되지 않아  
데이터 무결성이 깨짐.

#### 해결

- 완료된 Step을 Stack 구조로 관리
- LIFO 방식으로 보상 실행

---

## 8. 성과

- 데이터 정합성 보장
- 수동 복구 제거
- 장애 대응 시간 단축
- 재처리 자동화

---

## 9. 아키텍처

### Layered Architecture

- Application Layer  
  트랜잭션 흐름 제어

- Domain Layer  
  Step 및 보상 로직

- Infrastructure Layer  
  상태 저장 및 조회

---

## 10. 한계 및 개선 방향

### 현재 한계

- 단일 서비스 기준 설계
- 메시지 브로커 기반 이벤트 처리 미적용

### 개선 방향

- Saga 패턴 적용
- Kafka 기반 이벤트 드리븐 구조 전환
- 분산 환경에서의 최종 일관성 보장

---

## 11. 배운 점

- 분산 환경에서 ACID만으로는 정합성을 보장할 수 없다.
- 보상 트랜잭션의 핵심은 멱등성이다.
- 상태 기반 설계는 장애 대응력을 크게 향상시킨다.
